# 001 OpenClaw架设部署｜DocumentRules 文档编号规则｜v1.02

本文件定义 OpenClaw_vv 仓库内项目文档、版本化目录和角色包的命名与留存方式。

## 1. 项目分类目录

每个项目或文档分类使用独立编号：

```text
{编号}-{项目名分类}
```

当前示例：

```text
000-OpenClaw文档管理
001-OpenClaw规划设计
002-OpenClaw部署进度
003-OpenClaw事故经验
```

编号表示项目分类，不表示阅读顺序。阅读顺序由仓库根目录 `README.md` 维护。

## 2. 文档文件命名

目录内版本化文档使用：

```text
{English文档名中文文档名}-v{文档版本号}.md
```

示例：

```text
FinalDesign最终设计-v1.02.md
DeploymentPlan部署方案-v0.02.md
```

文档一级标题应包含项目编号、项目名、文档名和版本号。

## 3. 版本化功能目录命名

需要同时保存多个文件、可整体复制或部署的内容，使用版本化功能目录：

```text
{English目录名中文目录名}-v{目录版本号}/
```

示例：

```text
AgentCards角色卡-v0.02/
```

规则：

- 功能目录本身必须带版本号。
- 新建、试运行目录从 `v0.01` 开始。
- 小幅增补或修正递增 `0.01`，例如 `v0.01 → v0.02`。
- 已确认可长期使用时升为 `v1.0`。
- 结构明显变化时递增阶段版本，例如 `v1.0 → v1.1`。
- 新版本目录创建后，旧版本默认继续保留，并通过完整版本号区分。
- 当前使用版本必须由 README、设计文档或部署文档明确指向，避免多个版本同时被误认为最新版。
- 旧版本仅在薇明确决定清理时删除；需要降低误用风险时，可标记为历史版本或归档版本，但不自动删除。

## 4. 角色包目录命名

角色卡库中的每个可部署 Agent 使用独立、版本化角色包目录：

```text
{agent-id}-{正式角色名}-v{角色包版本号}/
```

示例：

```text
housekeeper-賈南風-v1.0/
ops-魚玄機-v0.01/
```

规则：

- `agent-id` 必须与 OpenClaw 实际配置使用的稳定 ID 一致。
- 正式角色名按仓库最终设计统一；当前 `housekeeper` 正式名称统一为繁体“賈南風”，其他说明文字使用简体中文。
- 角色包经过文案审查、权限审查并确认可直接部署后使用 `v1.0`。
- 尚在设计或试运行的角色包从 `v0.01` 开始。
- 角色包版本独立于角色卡库目录版本。
- 角色包内的 OpenClaw workspace 文件保持固定原生文件名，不在文件名中加入版本号，版本由父目录承担。
- 新角色包版本不得覆盖、改写或删除旧版本目录。
- 同一 Agent 可以保留多个历史版本，但只能有一个版本被索引和部署文档标记为当前版本。
- 清理历史角色包必须由薇明确决定，不能因创建新版本自动执行。

## 5. 角色包内部结构

可直接部署的角色包至少包含：

```text
{agent-id}-{正式角色名}-v{版本}/
├── README.md
├── IDENTITY.md
├── SOUL.md
├── AGENTS.md
├── USER.md
├── TOOLS.md
└── PERMISSIONS.md
```

其中：

- `README.md`：角色说明、版本、来源、部署与验证方法。
- `IDENTITY.md`、`SOUL.md`、`AGENTS.md`、`USER.md`、`TOOLS.md`：可直接复制到实际 Agent workspace 使用。
- `PERMISSIONS.md`：建议权限矩阵和必须由真实配置实现的限制，不直接复制为 OpenClaw 权限配置。
- 部署状态、已知限制和后续能力缺口可以使用独立版本化状态文档保存在对应角色包目录中，但不得混入五个 workspace 文件。

## 6. 版本号含义

| 版本 | 含义 |
| --- | --- |
| `0.01` | 新开发、新推进、新试运行内容的起始版本 |
| `0.02`、`0.03` | 试行、部署和纠错过程中的小步迭代 |
| `1.0` | 已定稿，可作为正式方案或直接使用的角色包 |
| `1.01`、`1.02` | 定稿后的少量文字或规则修正，不推翻主结构 |
| `1.1`、`1.2` | 定稿后的阶段性结构更新 |
| `2.0` | 主结构变化或重大重构 |

## 7. 禁止规则

- 不再使用 `00`、`10`、`20`、`30`、`40` 表示阅读顺序。
- 不使用无版本号的功能目录或角色包目录。
- 不在可部署 workspace 文件中使用“同上一版”“此处省略”等占位内容。
- 不通过复制目录制造两个名称相同但内容不同的“最新版本”。
- README、设计文档和部署说明引用角色卡时，必须指向明确版本路径。
- 不因创建新版本而自动删除旧版本。
- 不允许同一 Agent 的多个版本同时被标记为当前部署版本。

## 8. 版本留存原则

- 历史版本默认保留，用于回溯设计演进、部署差异和事故复盘。
- 当前版本通过根 README、角色卡库 README 和部署文档共同指向。
- 旧版本可以标记为 `historical`、`archived` 或“历史版本”，但仍保留原始内容。
- 只有薇明确决定清理后，才删除指定旧版本。
- 删除历史版本前，应先确认没有部署记录、事故复盘、链接或回滚流程仍依赖该版本。
