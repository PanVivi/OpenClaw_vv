# 001 OpenClaw架设部署｜DocumentRules 文档编号规则｜v1.0

本文件定义 OpenClaw_vv 仓库内项目文档的编号方式，避免再使用 `10/20/30/40` 这类排序编号。

## 1. 基本规则

每个任务项目必须有自己的独立项目编号。

当前项目编号：

```text
001 = OpenClaw架设部署
```

目录格式：

```text
{编号}-{项目名分类}
```

示例：

```text
001-OpenClaw规划设计
002-OpenClaw部署进度
003-OpenClaw事故经验
```

目录内文档文件名格式：

```text
{English文档名中文文档名}-{文档版本号}.md
```

示例：

```text
FinalDesign最终设计-v1.01.md
DeploymentPlan部署方案-v0.02.md
```

## 2. 版本规则

| 版本 | 含义 |
| --- | --- |
| `0.01` | 新开发、新推进、新试运行文档的起始版本 |
| `0.02`、`0.03` | 在试行、部署、纠错过程中逐步迭代 |
| `1.0` | 已定稿、已经确认可作为主干设定或正式可行方案 |
| `1.01`、`1.02` | 定稿后的少量内容更新，不推翻主结构 |
| `1.1`、`1.2` | 定稿后的阶段性大版本更新，通常涉及结构、流程或部署策略的明显变化 |
| `2.0` | 发生主结构变化或重大重构后的新正式版本 |

## 3. 当前仓库执行方式

- 主干设定、最终组织架构、AI 快速接手说明：当前使用 `1.01`。
- 部署方案、流程、进度、纠错、经验、实施路线：当前仍处于推进和验证阶段，已更新到 `0.02` 或保留原验证版本。
- 少量文字、命名、称呼、说明修正：版本号只增加 `0.01`，例如 `1.0 → 1.01`、`0.01 → 0.02`。
- 阶段性大版本更新：版本号增加 `0.1`，例如 `1.0 → 1.1`。
- 当某份部署或流程文档经过实际运行确认可长期使用时，再升为 `1.0`。

## 4. 禁止规则

- 不再使用 `00`、`10`、`20`、`30`、`40` 作为文件排序编号。
- 不用文件名前缀表达阅读顺序。
- 阅读顺序由 `README.md` 维护。
- 同一目录下不同主题可以有相同版本号，例如多个 `*-v0.02.md` 文档。

## 5. 文档标题规则

文档一级标题应包含编号、项目名、文档名和版本号：

```text
# 001 OpenClaw架设部署｜DeploymentPlan 部署方案｜v0.02
# 001 OpenClaw架设部署｜FinalDesign 最终设计｜v1.01
```
