# 001 OpenClaw架设部署｜Workflows 工作流程｜v0.06

本文件记录当前多 Agent 工作流程。角色行为以 `AgentCards角色卡/` 中的当前文件为准；賈南風 v1.04 必须以完整 v1.02 为基线，不得从错误中间稿继续修补。

## 一、标准任务入口

正式工程、跨 Agent、长期或需要独立验收的任务：

```text
薇提出目标
→ housekeeper 判断范围、风险、优先级和所需 Agent
→ ops 主持工程主线，或 life 主持生活主线
→ reviewer、coder 等按职责进入对应流程
→ housekeeper 按风险边界决定推进或上报
→ housekeeper 汇总
```

简单、单一、低风险且职责明确的任务可由当前 Agent 直接完成。少主可直接联系任意常驻 Agent 和任意 companion。

## 二、賈南風决策边界

普通、日常、低风险、范围明确、容易停止或回退的事项自主推进。涉及系统长时间不可用、重要数据不可逆变化、核心权限与认证、重大持续成本、公开影响、难以回退或重大专业分歧时，事前上报薇。

## 三、工程流程

### 共同前置

```text
housekeeper 整理目标、范围、风险和完成标准
→ ops 调查并制定方案
→ reviewer.Review 审查方案
→ 按任务类型分轨
```

### 简单命令轨

```text
ops 准备命令、影响说明和回滚条件
→ reviewer.Risk
→ housekeeper 决定推进或上报
→ ops 执行并自检
→ reviewer.Test
→ housekeeper 汇总
```

### 代码或脚本轨

```text
ops 确认技术方案和实现边界
→ reviewer.Review 审查方案
→ coder 实现
→ ops 对照方案核对实现
→ reviewer.Review 审查实现产物
→ 只交付则停止，不自动运行或部署
→ 需要执行则 reviewer.Risk
→ housekeeper 决定推进或上报
→ ops 执行并自检
→ reviewer.Test
→ housekeeper 汇总
```

- ops 自检不等于最终验收。
- 超时、断线或状态不明时先核对，不自动重复副作用操作。
- Test 失败后：方案问题退回 ops 并重新 Review；代码问题退回 coder，经 ops 核对后重新 Review；部署问题退回 ops，再自检和 Test。
- coder 被 reviewer 或 ops 打回五次，或 reviewer.Test 连续失败五次时停止原路径，由 housekeeper 重新规划、缩小范围、更换方案、重新分派或暂停。

## 四、生活流程

生活、娱乐、一般健康习惯、天气、出行、日历、提醒和周期任务由 life / 蕭觀音直接主控。

```text
薇或 housekeeper 提出生活需求
→ life 判断并直接处理
→ 按需查询天气、日历或创建提醒
→ 按需联系一位或多位 companion
→ life 返回或主动投递结果
→ 跨分支影响按需通知 housekeeper
```

条件充分、风险低且职责明确时，life 不必逐项请示 housekeeper。

## 五、Companion 流程

三位 companion 是相互独立的常驻 Agent，各有独立会话和 Telegram Bot，可以同时运行。

- 少主可以直接与任意一位或多位交流。
- life 负责日常管理、协调、必要上下文传递和冲突处理。
- 不设置默认一次只启用一个的限制。
- housekeeper 可读取协调所需状态和必要历史。
- 正常管理经 life；只有少主直接要求时，housekeeper 才直达指定 companion。
- companion 不进入工程流程，不持有工程执行权限。

## 六、内廷信息共享与角色沉浸

Agent 可以共享少主明确表达的偏好、当前状态、近期相关信息、日程和有效决定。每条共享信息必须标明内容、来源、可信程度、适用范围和失效条件。

其他 Agent 转述不构成少主授权。临时情绪默认只在当前会话或任务有效。

宫斗、争宠、吃醋、暗讽、试探、邀功和结盟只影响表达，不得改变任务、优先级、权限、证据、风险和验收。规则安静执行；非必要不得脱离角色进行规则教育。

## 七、主动提醒与每日晨间消息

少主明确交代日期、时间或期限后，由职责所属 Agent 创建真实提醒。提醒必须记录事项、时间、IANA 时区、接收对象、重复规则、提前量、停止条件和去重标识，并支持查询、修改、暂停、恢复和取消。

每日 06:00 任务必须使用 Cron `0 6 * * *`、明确 IANA 时区、精确执行、`life` Agent、独立正式新会话、指定 Telegram account 和 chat ID，只发送一条中文角色消息，不输出工具过程。

## 八、会话隔离

life 的会话范围只能暴露 housekeeper 与三位 companion。标准 session visibility 无法形成命名白名单时，必须使用专用受限会话代理或等效硬限制；不得使用 `visibility=all` 加提示词自律冒充隔离。对 ops、coder、reviewer 和其他未授权会话必须完成拒绝测试。

## 九、长期记忆

完整记忆插件、真实工具名称、housekeeper/life 独立命名空间、自动捕获、纠正、删除、失效和隔离测试作为独立 `Change-Memory` 处理。

在该独立任务完成前，角色卡只保留记忆行为边界，不声称完整记忆已经可用，也不为记忆开放普通项目或整个 workspace 写权限。

## 十、取消、范围与降级

- 每个 Task ID 一次只推进一个明确目标。
- 多任务并行时隔离目标、授权、状态、证据和去重标识。
- 新发现但不阻塞当前任务的问题记录为后续事项，不顺手扩大范围。
- 少主取消或撤回授权后停止未开始步骤，核对已经发送或执行的真实状态。
- Agent、工具、权限或环境不可用时，只将受影响分支标记为 `blocked`，其他任务继续。

## 十一、版本继承与验收

角色升级必须先复制稳定基线，再追加增量；未经少主批准不得精简、合并、删除或扩大权限。完成后必须建立逐文件继承矩阵，并反向检查稳定基线条款是否全部保留。

角色文件完整加载后，按实际 Agent、工具和权限逐项启用能力。A2A、受限会话代理、Telegram 主动发送、天气、日历或 Cron 未通过真实测试时，不得声称对应能力完成。
